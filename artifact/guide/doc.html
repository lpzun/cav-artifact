1. Design Description

An overview of PAT

The above figure demonstrates the architecture of our tool. It contains two components: PAT and QuTL MCer.

  • PAT implements the core idea of our approach (Algorithm 1, Section 4 of the paper). It contains two main procedures: the procedure of observation sequence construction using list abstraction, and that of convergence detector using the partial abstract transformer.

  • QuTL MCer (QuTL model checker) implements the QuTL model checking algorithms (Section 5 of the paper). When enable, it is used to filter the suprious abstract states in the process of convergence detection. It contains two main procedures: a concrete MCer is called on concrete states in the process of sequence construction. Its main purpose is to verify whether the input invariant is valid or not. If the concrete MCer reports a violation, then the user has to rewritten the invariant. Otherwise, the invariant will be fed to abstract MCer to filter infeasible abstract states -- the ones does not satisfy the invariant.

Usage: To use our tool, the users first need to compile a P program, which is done via P compiler (See tutorial and Section 6 of the paper for more details). Then, our tool will run the verification on the compiled P program. Our tool provides rich features to perform verification. Two main features are presented as follows:

  • /queue-prefix:p:: to verify P program by specifying an unabstracted prefix of queues. An advantage of this approach is that it is fully automatic.

  • /qutl:formula :: verify P program WITH QuTL MCer.
Of course, we can even combine above two features together, namely we verify a P program by inputing an invariant, and meanwhile specify an unabstracted prefix.
